### 3.2 核心代码变更(`proxy_model.go`)

**改前**(第 132-133 行):

```go
// 3. 绑定工具
p.toolBinder.BindToModel(&model.ChatModel)
```

问题:`&model.ChatModel` 是接口字段的**地址**,不是接口值本身。`BindToModel` 内部用 `model.(ToolBindable)` 做类型断言时,由于类型不匹配(`*einoModel.ChatModel` 不实现 `ToolBindable`),**永远落入 fallback 分支**,日志输出 "model does not support BindTools" 后静默跳过。于是后续 `executor(model)` 里的 `m.ChatModel.Generate(...)` 发送的请求没有 tools 字段。

**改后**:

```go
// 3. 将已登记的工具绑定到本次使用的模型实例
if toolInfos := p.toolBinder.GetTools(); len(toolInfos) > 0 {
    if tcm, ok := model.ChatModel.(einoModel.ToolCallingChatModel); ok {
        withTools, bindErr := tcm.WithTools(toolInfos)
        if bindErr != nil {
            klog.Warningf("ProxyChatModel: failed to bind tools to model [%s]: %v", model.APIKeyName, bindErr)
        } else {
            model = &ModelWithMetadata{
                ChatModel:  withTools,
                APIKeyName: model.APIKeyName,
                APIKeyID:   model.APIKeyID,
                LLMModel:   model.LLMModel,
            }
            klog.V(6).Infof("ProxyChatModel: bound %d tools to model [%s]", len(toolInfos), model.APIKeyName)
        }
    } else {
        klog.V(6).Infof("ProxyChatModel: model [%s] is not ToolCallingChatModel, tools not bound", model.APIKeyName)
    }
}
```

要点:
1. 用 Eino 官方推荐的 `WithTools` 接口(返回新实例,避免状态污染)
2. 不改 `toolBinder` 行为,只修补 `Generate/Stream` 调用链上"忘了取工具来用"这一步
3. 失败时 graceful fallback 到原模型,不让修复反而引入崩溃

### 3.3 为什么要把 `ModelWithMetadata` 改成 `BaseChatModel`

`einoModel.ChatModel` 在 Eino v0.7+ 已经被标记为 deprecated,并且**不是 `ToolCallingChatModel` 的超集**(前者要求 `BindTools`,后者要求 `WithTools`,两者都只继承 `BaseChatModel`)。

我要把 `withTools` 的返回值(`ToolCallingChatModel`)存回去给后续 `executor` 调用,而 `ModelWithMetadata` 的字段类型如果还是 `ChatModel`,Go 编译器会拒绝赋值(`ToolCallingChatModel` missing `BindTools`)。

`BaseChatModel` 只要求 `Generate/Stream`,两个子类型都能赋值,而后续代码(`executor`)只调用 `Generate/Stream`,完全够用。这是最小破坏性的接口修正。
