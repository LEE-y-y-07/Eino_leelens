## 一、调用链路梳理

### 1.1 正常的 Agent 执行预期流程

```
adk.NewRunner().Run()
  → ChatModelAgent 内部 Loop
    → chatModel.BindTools(toolInfos)     // 登记工具
    → chatModel.Generate(ctx, messages)   // 每轮生成:请求里应包含 tools 字段
      → 若模型返回 tool_calls:
        → 执行对应 tool(list_dir / read_file / ...)
        → 把 tool_result 追加到 messages
        → 继续下一轮 Generate
    → 直到模型不再返回 tool_calls 为止
```

### 1.2 本项目对应实现

`internal/pkg/adkagents/manager.go:238-345` 的 `createADKAgent()`:

当 YAML 里 `model: ""` 且没有 `models:` 列表时,走第 266-273 行:

```go
} else if m.enhancedModelProvider != nil {
    chatModel = NewProxyChatModel(m.enhancedModelProvider, nil)
}
```

这里的 `chatModel` 是一个 `*ProxyChatModel`(项目自定义的动态代理,用来做模型池轮询、限流降级)。它作为 `config.Model` 被传给 `adk.NewChatModelAgent`。

## 二、Bug 根因

**`ProxyChatModel` 内部在调用"池中实际模型"的 `Generate/Stream` 时,没有把之前存进 `toolBinder` 的工具传递下去。**

- Eino ADK 在 Agent 执行前调用 `p.BindTools(tools)` —— 工具**只是存进了 toolBinder 字段**
- 真正发起 LLM 请求是通过 `executeWithRetry` 里的:
  ```go
  return m.ChatModel.Generate(ctx, input, opts...)
  ```
- 这里的 `m.ChatModel` 是由 `provider.createOpenAIChatModel()` / `createClaudeChatModel()` **新创建的裸实例**,它没有调用过 `BindTools` / `WithTools`
- 所以这个裸实例发出的请求里 **`tools` 字段为空**
- 模型看不到任何工具,只能凭对话里的文字去"想象"

### 顺带说一句

`tool_binder.go:51` 的 `ToolBinder: model does not support BindTools` 警告也能佐证:ADK 框架的某条辅助路径尝试把工具绑到模型上,但因为 `m.ChatModel` 是裸 OpenAI 实例、不实现本项目自定义的 `ToolBindable` 接口,于是静默跳过。日志虽然是 `V(6)` 的 Info 级,但这就是 bug 在日志里的蛛丝马迹。

## 三、为什么之前没被发现

### 假设 1:原开发环境曾配置 `model: "xxx"` 具体名称走第二分支

`manager.go` 的第二分支:

```go
} else if def.Model != "" && m.enhancedModelProvider != nil {
    cm, err := m.enhancedModelProvider.GetModel(def.Model)
    chatModel, ok = cm.(model.ToolCallingChatModel)
    // chatModel 是 openai.ChatModel 或 claude.ChatModel 原生实例
    // 它们原生实现了 ToolCallingChatModel,Eino 能正确绑定工具
}
```

如果过去 YAML 都用了具体模型名,那条路径工具绑定正常。但**一旦切到"第三分支(ProxyChatModel,自动池)",工具就消失了**。

### 假设 2:测试时用的仓库很小,LLM 幻觉"看起来很像",躲过人肉验证

这次用 9 模块的 Spring AI 仓库暴露得很彻底 —— 模型必须提到具体模块名才能显得真实,幻觉无处藏身。

### 假设 3:即便运行了,也没人检查 `tool_count` 这条日志

`[EinoCallback] Model 输入 tool_count=0` 藏在 klog `V(6)` 详细日志里,日常运行看不到。

## 四、修复方案

### 核心改动:在 `executeWithRetry` 调用内部模型前,绑上 `toolBinder` 里存的工具

```go
// 在 executeWithRetry 拿到池中 model 之后、调用 executor 之前:
var toolModel model.ToolCallingChatModel
if tools := p.toolBinder.GetTools(); len(tools) > 0 {
    if tcm, ok := m.ChatModel.(model.ToolCallingChatModel); ok {
        withTools, err := tcm.WithTools(tools)
        if err == nil {
            // 用带工具的新实例替换调用目标
            m = &ModelWithMetadata{
                ChatModel:  withTools,
                APIKeyName: m.APIKeyName,
                APIKeyID:   m.APIKeyID,
                LLMModel:   m.LLMModel,
            }
        }
    }
}
```

然后 `executor(m)` 用的就是带工具的实例。

### 为什么用 `WithTools` 而不是 `BindTools`?

- Eino 的 `ToolCallingChatModel` 接口规定 `WithTools` 返回**新实例**,不污染原模型
- 多个 Agent 可能共用同一个 `provider`,如果我们用 `BindTools` 就地修改,会互相串味
- `WithTools` 语义更干净

### 不修改 `BindTools` / `WithTools` 本身,只修补 `Generate`/`Stream` 调用点

因为 `BindTools(tools)` 存入 `toolBinder` 的行为是对的,它是"记住用户想要绑哪些工具",真正的问题是**忘记在每次实际调用时把这笔记读出来用**。

## 五、影响面评估

### 修改的文件
- `internal/pkg/adkagents/proxy_model.go`:`executeWithRetry` 内新增 ~15 行

### 可能的副作用
- 每次 `Generate/Stream` 调用多做一次类型断言和 `WithTools`,**性能可忽略**(毫秒级内存操作)
- 某些模型的 `WithTools` 可能失败(返回 error) —— 我们 graceful fallback 到原实例,不要让 bug 修复反而导致不能执行
- `ModelWithMetadata` 的其他字段(APIKeyID 等)要完整复制,避免 `recordUsage` 统计丢信息

### 单测影响
- 现有 `proxy_model` 没有单测(查了下 `internal/pkg/adkagents/` 下相关 `_test.go`),需要补一个最小集
